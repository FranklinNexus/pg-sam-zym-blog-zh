# 受欢迎的秘诀

> 原文：<https://paulgraham.com/popular.html>
> 来源：[paulgraham.com](https://paulgraham.com/)
> 英文标题：Being Popular
> 年代：2001 年 5 月

（本文最初是为一种新语言写的商业计划书。因此它略过了——因为默认已具备——一门好编程语言最重要的特性：极其强大的抽象能力。）

### 开篇问题

我有个朋友曾告诉一位杰出的操作系统专家，他想设计一门真正好的编程语言。专家说，这是浪费时间：编程语言会不会流行，并不取决于语言本身好不好；所以无论他设计得多出色，都没人会用。至少，他自己设计的那门语言就是这么回事。

那么，究竟是什么让一门语言流行？流行的语言配得上它们的流行吗？定义一门好编程语言值得吗？又该怎么做？

我认为答案在于观察黑客，弄清他们想要什么。编程语言是为黑客设计的；一门语言作为编程语言（而不是指称语义或编译器设计的练习）好不好，当且仅当黑客喜欢它。

### 流行的机制

确实，大多数人并不是根据语言的优劣来选语言的。多数程序员是被别人指定用什么。但我以为，这类外部因素对流行度的影响，并没有人们常想的那么大。更大的问题在于：黑客心目中的好语言，和多数语言设计者的想法并不一致。

在这两者之间，黑客的意见才重要。编程语言不是定理，而是给人用的工具，必须像鞋子合脚一样，贴合人的长处与弱点。要是穿上就夹脚，再像雕塑一样优雅，也是坏鞋。

也许多数程序员分不清好语言和坏语言。但别的工具也一样。这并不意味着设计好语言是浪费时间。[专家级黑客](https://paulgraham.com/design.html)一眼能认出好语言，并且会用。专家黑客固然是极少数，但这极少数人写出了所有好软件；他们的影响力如此之大，其他程序员往往会跟着用他们用的语言。而且常常不只是影响，而是命令：专家黑客往往正是那些当老板或导师、指定别人用什么语言的人。

决定语言相对流行度的，不只有专家黑客的意见——遗留软件（Cobol）和炒作（Ada、Java）也有作用——但我认为，从长远看，专家黑客是最强力量。只要有了初始临界质量和足够时间，一门语言大概会流行到它应得的程度。流行还会进一步把好语言与坏语言拉开，因为真实用户的反馈总会带来改进。看看任何流行语言一生中的变化：Perl 和 Fortran 是极端例子，连 Lisp 也变了很多。比如 Lisp 1.5 没有宏；这些功能是 MIT 的黑客们用 Lisp 写真实程序几年后才演化出来的。[1]

所以，不管一门语言是否必须好才能流行，我认为它必须流行才能好，而且必须保持流行才能保持好。编程语言的艺术状态不会静止。然而今天的 Lisp 大体上还是 1980 年代中期 MIT 那一版，因为那是 Lisp 最后一次拥有足够大、足够苛刻的用户群。

当然，黑客得先知道一门语言才能用。他们怎么知道？从别的黑客那里。但总得有一小撮黑客先用，别人才听得到。我在想，这撮人要多大才算临界质量？凭直觉，我会说二十个。如果一门语言有二十个独立用户——二十个自己决定要用它的人——我会认为它已经是「真语言」了。

走到这一步肯定不容易。从零到二十，可能比从二十到一千还难。拿到最初这二十个用户，最好的办法也许是木马战术：给人们一个他们想要的应用，而应用恰好是用新语言写的。

### 外部因素

先承认一个确实影响流行的外部因素：要流行，一门语言必须成为某个流行系统的脚本语言。Fortran 和 Cobol 是早期 IBM 主机的脚本语言。C 是 Unix 的脚本语言，后来 Perl 也是。Tcl 是 Tk 的脚本语言。Java 和 JavaScript 则旨在成为浏览器的脚本语言。

Lisp 没有大规模流行，部分原因是它不是某个大规模流行系统的脚本语言。它保留的那点流行度，要追溯到 1960、1970 年代——那时它是 MIT 的脚本语言。那个时代许多伟大程序员都与 MIT 有过交集。1970 年代初、C 出现之前，MIT 的 Lisp 方言 MacLisp，是严肃黑客愿意用的少数语言之一。

今天 Lisp 是两个中等流行系统的脚本语言：Emacs 和 AutoCAD。因此我怀疑，如今大部分 Lisp 编程是在 Emacs Lisp 或 AutoLisp 里完成的。

编程语言不是孤立存在的。编程是及物动词——黑客通常是在折腾某样东西——实践中，语言是相对于所折腾的对象来评判的。所以，若想设计一门流行语言，你要么提供的不仅是一门语言，要么就得设计一门能取代某现有系统脚本语言的语言。

Common Lisp 不流行，部分因为它是孤儿。它最初确实附带一个可以折腾的系统：Lisp 机器。但 1980 年代，Lisp 机器（以及并行机）被越来越强的通用处理器碾过去了。如果 Common Lisp 是 Unix 的好脚本语言，也许还能保持流行。可惜，它是 Unix 上糟透了的脚本语言。

一种说法是，语言不能单凭自身优点被评判。另一种看法是，除非同时是某系统的脚本语言，否则它根本不算编程语言。这只有在你没料到的时候才会觉得不公平。我觉得，这和要求语言有实现一样，不算不公平；这就是语言的一部分。

当然，语言还需要好的实现，而且必须免费。公司会为软件付钱，个人黑客不会；而你要吸引的是黑客。

语言还需要有一本书。书要薄、写得好、例子好。K&R（《C 程序设计语言》）是典范。眼下我差不多可以说，一门语言得有一本 O'Reilly 出的书，这才算进入黑客的视野。

还应有在线文档。事实上，书可以从在线文档起步。但纸质书还没过时：格式方便，出版商的事实审查虽不完美，却是有用的筛选。书店仍是了解新语言的重要场所之一。

### 简洁

既然你能提供任何语言所需的三样东西——免费实现、一本书、以及可以折腾的对象——怎样做出黑客喜欢的语言？

黑客喜欢简洁。黑客很懒，就像数学家和现代主义建筑师一样懒：他们讨厌一切多余。说黑客动笔前（至少潜意识里）会根据总共要敲多少字符来选语言，并不离谱。即便黑客未必真这么想，语言设计者最好也按这个假设来设计。

试图用冗长、像英语的表达式来「照顾用户」，是个错误。Cobol 因此臭名昭著。让黑客写

```
add x to y giving z
```

而不是

```
z = x + y
```

会被视为对智商的侮辱，甚至是对上帝的亵渎。

有人说 Lisp 该用 first 和 rest 代替 car 和 cdr，程序更易读。也许头几个小时是。但黑客很快就能学会：car 是列表首元素，cdr 是其余部分。用 first 和 rest 要多敲 50% 的键。而且长度不同，连续几行调用时参数对不齐——car 和 cdr 常常正是这样排列的。我发现代码在页面上如何对齐非常重要。Lisp 用比例字体显示时我几乎读不下去，朋友说别的语言也一样。

简洁是强类型语言的短板。其他条件相同时，没人愿意在程序开头写一堆声明。能隐式表达的，就该隐式。

单个记号也要短。Perl 和 Common Lisp 在这个问题上处于两极：Perl 可以写得几乎像谜语一样密，Common Lisp 内建操作符的名字则长得可笑。Common Lisp 的设计者大概指望用户有能自动输入长名字的编辑器。但长名字的成本不只是打字，还有阅读成本，以及占屏幕空间的成本。

### 开放与可改造

对黑客来说，有一件事比简洁更重要：能按自己的意愿做事。编程语言史上，令人惊讶的是，设计者花了大量精力阻止程序员做「不合适」的事。这是极其冒昧的计划。设计者怎么知道程序员将来需要什么？我认为，设计者应把目标用户当作天才——会做他们从未预料到的事——而不是需要被保护的笨蛋。笨蛋反正会在自己脚上开枪：你也许能阻止他引用别的包里的变量，却阻止不了他写一个设计糟糕、解决错误问题、还要做一辈子的程序。

好程序员常常想做危险、不太「体面」的事。所谓不体面，是指突破语言试图呈现的语义外壳：比如拿到某个高级抽象的内部表示。黑客喜欢折腾，折腾就是钻进内部、二次猜测最初的设计者。

允许别人二次猜测你。任何工具都会被以你没想过的方式使用，高度表意的编程语言尤其如此。许多黑客会以你从未设想的方式改动你的语义模型。我说：随他们去；在不危及垃圾回收等运行时系统的前提下，尽量让程序员接触内部。

在 Common Lisp 里，我常想遍历结构体的字段——比如梳出对已删除对象的引用，或找出未初始化字段。我知道结构体底层只是向量。然而我写不出一个能在任意结构体上调用的通用函数，只能按字段名访问，因为结构体「应该」表示的就是那个意思。

黑客在大程序里也许只想颠覆设计模型一两次。但能不能做到，差别巨大。这不只是解决问题，也是一种乐趣。黑客共享外科医生在黏腻内脏里探查的秘密快感，以及少年挤痘痘的秘密快感。[2] 至少对男孩来说，某些恐怖有种迷人吸引力。《Maxim》杂志每年出一本图册，混排美女照和可怕事故，他们懂自己的读者。

历史上，Lisp 很擅长让黑客为所欲为。Common Lisp 的政治正确是反常。早期 Lisp 让你接触一切；幸运的是，宏在一定程度上保留了这种精神。能对源代码做任意变换，多么美妙。

经典宏是真正的黑客工具——简单、强大、危险。作用极易理解：对宏的参数调用一个函数，返回值插入宏调用处。卫生宏则体现相反原则：尽量让你弄不懂它们在干什么。我从没听过谁能用一句话解释卫生宏。它们是「替程序员决定该要什么」的经典危险例子。卫生宏想保护我不被变量捕获，而有时我恰恰想在某些宏里捕获变量。

真正好的语言应该既干净又脏：设计干净，核心操作符少而正交、理解透彻；同时又脏，因为允许黑客随意改造。C 如此，早期 Lisp 也如此。真正的黑客语言总带一点放荡不羁。

好语言应有让说「软件工程」的人摇头否定的特性。另一端是 Ada、Pascal 这类端庄的模范，适合教学，除此之外用处不大。

### 一次性程序

要吸引黑客，语言必须适合写他们想写的程序。这意味着——也许出人意料——它必须适合写「一次性程序」。

一次性程序是为有限任务快速写的程序：自动化系统管理、为仿真生成测试数据、把数据从一种格式转成另一种。令人惊讶的是，就像二战期间美国许多大学建的「临时」建筑一样，它们往往不会被扔掉。许多演化成真正的程序，有真功能、真用户。

我隐约觉得，最好的大程序往往这样开始，而不是像胡佛大坝那样一开始就按大型工程来设计。从零造大东西是可怕的。项目太大，人会被压垮：要么陷住，要么结果生硬呆板——像购物中心而非真市中心，像巴西利亚而非罗马，像 Ada 而非 C。

另一条路是从一次性程序起步，不断改进。这种方式不那么吓人，设计也受益于演化。我想细查的话，会发现多数大程序就是这么来的。而且这样演化出来的程序，多半仍用最初的语言写——程序很少被移植，除非出于政治原因。因此，矛盾的是：若想让语言用于大系统，就得让它适合写一次性程序，因为大系统正是从一次性程序长出来的。

Perl 是显著例子。它不仅为一次性程序而设计，本身几乎就是一次性程序：起初是生成报告的工具集合，人们用它写的一次性程序越来越大，才演化成语言。直到 Perl 5（如果算的话）才适合写严肃程序，那时它早已大规模流行。

什么让语言适合一次性程序？首先，必须随手可得。一次性程序是你指望一小时内写完的。语言大概得已经装在你用的机器上，不能是用了才装。它得在那儿。C 在那儿，因为随操作系统提供。Perl 在那儿，因为最初是系统管理员工具，你的管理员已经装好了。

可得还不只是已安装。带命令行的交互式语言，比须单独编译运行的更易得。流行语言应是交互式的，且启动要快。

一次性程序还需要简洁。简洁对黑客总有吸引力，在预期一小时内交差的程序里尤其如此。

### 库与生态

简洁的极致，是程序已经写好，你只消调用。这就说到我认为会越来越重要的特性：库函数。Perl 能赢，很大程度上是因为它有大量处理字符串的库。这类库对一次性程序尤其重要——它们常最初是为转换或提取数据而写。许多 Perl 程序大概一开始只是把几个库调用拼在一起。

我认为，未来五十年编程语言的进展，很大程度在库函数上。未来语言的库，会和核心语言一样精心设计。语言设计将不再主要是强类型还是弱类型、面向对象还是函数式之类，而是如何设计伟大的库。爱琢磨类型系统的设计者可能会发怵：这几乎像在写应用！可惜。语言是为程序员服务的，程序员需要的是库。

设计好库很难。不只是写很多代码。库太大时，有时找函数比自己写还慢。库应用一小套正交操作符来设计，就像核心语言一样。程序员应能猜出哪个库调用能完成所需。

库是 Common Lisp 的短板。处理字符串的只有基础库，与操作系统交互的几乎没有。出于历史原因，Common Lisp 假装操作系统不存在。无法与操作系统对话，就很难只用内建操作符写严肃程序；还得用一些实现相关的变通手段，而实践中这些往往给不全你想要的。如果 Common Lisp 有强大的字符串库和良好的操作系统支持，黑客对 Lisp 的评价会高得多。

### 语法

一门具有 Lisp 语法——更准确说，缺乏语法——的语言，能流行吗？我不知道。但我以为，语法不是 Lisp 目前不流行的主因。Common Lisp 有比陌生语法更严重的问题。我认识几个习惯前缀语法、却默认用 Perl 的程序员，因为 Perl 字符串库强、能跟操作系统说话。

前缀表示法有两个可能的问题：程序员不熟悉，以及不够紧凑。Lisp 界的传统看法是第一个才是要害。我不那么确定。是的，前缀会让普通程序员恐慌；但普通程序员的意见不重要。语言流不流行，看专家黑客怎么想；我认为专家黑客能应付前缀。Perl 语法可以相当难懂，却没挡住 Perl 的流行。若有帮助，也许还助长了 Perl 文化。

更严重的是前缀的冗长。对专家黑客，这确实是问题。没人想写 `(aref a x y)`，当可以写 `a[x,y]` 时。

这个具体情形有变通：若把数据结构看作索引上的函数，可以写 `(a x y)`，比 Perl 形式还短。类似技巧也许能缩短其他表达式。

让缩进有意义，可以去掉（或可选）许多括号。程序员本来就这么读代码：缩进说一套、定界符说另一套时，我们跟缩进走。把缩进当语法能消除这类常见错误，也让程序更短。

有时中缀更易读，数学表达式尤其如此。我用 Lisp 写程序一辈子，仍不觉得前缀数学式自然。但在生成代码时，能接受任意数量参数的操作符很方便。所以若用中缀，最好实现成某种读取宏。

我不认为应宗教式地反对给 Lisp 引入语法，只要它能以清晰的方式翻译成底层 s 表达式。Lisp 里已有不少语法。引入更多未必坏事，只要没人被强制使用。Common Lisp 里有些定界符预留给语言，说明至少部分设计者打算将来加更多语法。

Common Lisp 里最不像 Lisp 的语法之一在 format 字符串里：format 自成一门语言，而且不是 Lisp。若有给 Lisp 加语法的计划，format 说明符也许能纳入。若宏能像生成其他代码一样生成 format 说明符，会是好事。

一位著名 Lisp 黑客告诉我，他的 CLTL 总是翻开在 format 那一节。我也是。这也许说明有改进空间，也可能说明程序 I/O 很多。

### 效率

好语言，众所周知，应生成快速代码。但实践中，我认为快速代码主要不来自语言设计里的某些特性。Knuth 早就指出，速度只在少数关键瓶颈处重要；许多程序员后来也观察到，我们常会误判瓶颈在哪。

因此实践中，获得快速代码靠非常好的性能分析器，而不是比如一开始就强类型。你不必知道程序里每次调用的每个参数类型；你需要能在瓶颈处声明类型。更重要的是，你得能找到瓶颈在哪。

人们对 Lisp 的一大抱怨是，很难知道什么开销大。这也许属实，也许对高度抽象的语言不可避免。无论如何，我认为好的性能分析能很大程度缓解：你很快会知道什么贵。

部分问题是社会性的。语言设计者爱写快速编译器——那是衡量技艺的方式。他们把性能分析器当附加品。但实践中，好的性能分析器也许比生成快速代码的编译器更能提高实际程序的速度。这里设计者又和用户有点脱节：他们非常擅长解决稍微不对的问题。

也许该有「主动式」性能分析器——把性能数据推给程序员，而不是等他来问。比如编辑源码时，编辑器用红色标出瓶颈。另一种办法是以某种方式呈现运行中程序的状态。这对服务器应用尤其有效——有很多运行中的程序可看。主动式性能分析器可以图形化显示内存里正在发生什么，甚至用声音告诉你程序在干什么。

声音是问题的好提示。我曾工作的地方有一块大表盘，显示 Web 服务器状态。指针由小伺服电机驱动，转动时有轻微噪音。我从工位看不到表盘，却能凭声音立刻知道哪台服务器出了问题。

甚至也许能写性能分析器，自动检测低效算法。某些内存访问模式最终成为坏算法的明显标志，我不会惊讶。若计算机里有个小人在执行我们的程序，他的工作经历大概会像联邦政府雇员一样漫长而哀怨。我常觉得自己在让处理器做很多无用功，却一直没有好办法看它到底在干什么。

如今许多 Lisp 编译成字节码，再由解释器执行。这通常是为了移植实现，但也可以是语言特性。也许该把字节码定为语言官方部分，允许程序员在瓶颈处写内联字节码；这样优化也可移植。

终端用户感知的「速度」性质可能在变。随着服务器应用兴起，越来越多程序可能受 I/O 限制。让输入输出变快会值得。语言可以用直接措施帮忙，比如简单快速的格式化输出；也可以用深层结构变化，比如缓存和持久化对象。

用户关心响应时间。但另一类效率会越来越重要：每个处理器能支持多少并发用户。近期许多有趣应用将是服务器应用；对托管这类应用的公司，每服务器用户数是关键——在提供服务器应用的资本成本里，这是除数。

多年来，效率在多数终端应用里不太要紧。开发者可以假设每个用户桌上有一台越来越强的处理器；按帕金森定律，软件会膨胀去占满资源。服务器应用里这会变：硬件和软件将一起提供。对提供服务器应用的公司，每服务器能支持多少用户，对利润影响极大。

有些应用里处理器是瓶颈，执行速度是最该优化的事。但常常是内存：并发用户数取决于每个用户数据占多少内存。语言也能帮忙：好的线程支持让所有用户共享一个堆；持久化对象和/或语言级懒加载也可能有用。

### 时间与增长形态

流行语言需要的最后一种成分是时间。没人愿意用可能消失的语言写程序——而许多语言确实消失了。所以多数黑客会等一门语言存在几年，才考虑使用。

发明奇妙新事物的人常对此感到惊讶，但要把信息传给人，就需要时间。我有个朋友很少在第一次被请求时就做事。他知道人有时会要后来并不想要的东西。为避免浪费时间，他通常等到第三、四次被请求才动；到那时，问的人也许相当恼火，但至少他们是真想要。

多数人对听到的新事物也会做类似过滤：往往要听到十次才开始关注。这完全合理：多数热门新东西最终是浪费时间，然后消失。我拖延学 VRML，结果根本不用学。

所以任何发明新事物的人都得预期：重复自己的信息许多年，人们才开始明白。我们写了据我所知第一个基于 Web 服务器的应用，花了好几年才让人明白它不必下载。不是他们蠢，而是他们把我们过滤掉了。

好消息是，简单重复能解决问题。你只要不断讲你的故事，人们最终会开始听。不是人们注意到你在场才关注，而是注意到你还在场。

幸好获得动量通常需要时间。多数技术——尤其是编程语言——首次推出后还会大幅演化。对新技术，没有什么比几年里只被少数早期采用者使用更好。早期采用者老练、要求高，很快会冲掉你技术里剩下的缺陷。用户少时，你能和每个用户保持密切接触。早期采用者在你改进系统时，即使造成一些破坏，也会宽容。

新技术有两种引入方式：有机增长，和闪电式推广。有机增长的典型是经典的车库创业：几个家伙在默默无闻中开发新技术，几乎不做营销推出，最初只有极少数（极其忠诚的）用户；他们继续改进，用户群靠口碑增长，不知不觉就大了。

另一种是闪电式推广，典型是 VC 支持、大规模营销的初创：匆忙开发产品，大张旗鼓发布，立刻（希望）拥有大量用户。

车库创业的人常羡慕闪电式的人。后者光鲜、自信、受 VC 尊重，买得起最好的一切，发布公关还让他们成了名人。车库团队觉得自己穷、没人爱。但我认为他们常误读自己的处境：相比闪电式，有机增长往往孕育更好的技术和更富有的创始人。看今天主导的技术，多数是有机长出来的。

这模式不只适用于公司，也适用于资助研究：Multics 和 Common Lisp 是闪电式项目，Unix 和 MacLisp 是有机增长项目。

### 重新设计

E. B. White 写过：「最好的写作是重写。」每个好作家都知道，软件亦然。设计中最重要的部分是重新设计。编程语言尤其重新设计得不够。

写好软件，必须同时在脑子里保持两种对立想法：年轻黑客对自己能力的天真自信，以及老手的不信任。你得用一半大脑想「[这有多难呢？](http://www.trevorblackwell.com/)」，另一半想「[这永远不可能](http://www.pdos.lcs.mit.edu/~rtm/)」。

诀窍是认识到这里没有真矛盾：你要对两件事分别乐观和怀疑——对解决问题的可能性乐观，对眼下方案的价值怀疑。

做得好的人常觉得自己在做的东西不好。别人看他们成果满惊讶，创作者满担忧。这不是巧合：正是担忧让作品好。

若希望与担忧保持平衡，它们会像两条腿蹬自行车一样推动项目。创新两周期引擎的第一阶段：你因相信自己能解决问题而猛干。第二阶段：清晨冷光下审视成果，清楚看到所有缺陷。只要批判精神不超过希望，你就能看着显然不完整的系统，想「剩下这点有多难呢？」，从而继续循环。

保持两股力量平衡很难。年轻黑客乐观占上风：做出东西觉得棒，就不再改。老年黑客怀疑占上风：不敢接雄心项目。

任何能让重新设计循环继续的事都是好的。散文可以反复改到满意。软件通常重新设计得不够。散文有读者，软件有用户。作家重写一篇文章，读过旧版的人不太可能抱怨：新不兼容破坏了他们的思路。

用户是双刃剑：能帮你改进语言，也能阻止你改进。所以要谨慎选择用户，慢慢增加数量。拥有用户像优化：明智的做法是推迟。而且一般说来，任何时候你能改的比你想的更多。引入变化像撕创可贴：疼几乎在你感到的瞬间就成了记忆。

人人都知道委员会设计语言不是好主意，委员会产出坏设计。但我认为委员会最大危险是妨碍重新设计。引入变化太费劲，没人愿费心。委员会定下来的往往就僵住，即使多数成员并不喜欢。

甚至两个人的「委员会」也会妨碍重新设计，尤其两人写的软件之间的接口：改接口必须双方同时同意，于是接口往往不变——而接口常常是系统里最临时凑合的部分。

一种解法也许是把系统设计成接口横向而非纵向——模块总是垂直堆叠的抽象层。于是接口倾向于归某一层所有。两层中较低的一层，要么是上层所用的语言（下层拥有接口），要么是附庸（接口由上层定）。

### Lisp 的出路

这一切意味着，新 Lisp 仍有希望。任何给黑客所需的语言都有希望，包括 Lisp。我们可能错在以为黑客会被 Lisp 的怪异吓跑。这种安慰性幻觉也许让我们没看清 Lisp——至少是 Common Lisp——的真正问题：它不适合黑客想干的事。黑客语言需要强大的库和可以折腾的对象；Common Lisp 两者皆无。黑客语言应简练且可随意改造；Common Lisp 不是。

好消息是，问题不在 Lisp，而在 Common Lisp。若能开发一门真正属于黑客的新 Lisp，我认为黑客会用。他们会用能完成工作的任何语言。我们只需让新 Lisp 在某些重要任务上比别的语言更好。

历史有些鼓励：一代接一代的新语言从 Lisp 拿走越来越多特性。再抄下去，抄出来的就是 Lisp。最新热门 Python，是加了中缀、去了宏的稀释版 Lisp。新 Lisp 会是这一演化的自然一步。

我有时想，营销上不妨叫它「改进版 Python」——听起来比 Lisp 时髦。对许多人，Lisp 是慢吞吞的 AI 语言、满屏括号。Fritz Kunze 的官方传记小心避开 L 字。但我猜我们不应害怕把新 Lisp 叫 Lisp。Lisp 在顶尖黑客中仍有潜在的尊重——比如上过 6.001 并真懂的人。而你要赢的正是这些用户。

Eric Raymond 在《如何成为黑客》里把 Lisp 比作拉丁语或希腊语——应学作智力训练，尽管你不会真的用它：

> 学 Lisp 值得，因为当你终于搞懂时，会有深刻的启蒙体验；即使你一辈子不常用 Lisp 本身，这种体验也会让你此后成为更好的程序员。

若我不懂 Lisp，读到这会起疑：若一种语言能让我成为更好的程序员，若这话有任何意义，就意味着它对编程更好——而 Eric 说的正是这个意思。

只要这种想法还在，我认为黑客会对新 Lisp 足够开放，即使仍叫 Lisp。但这 Lisp 必须是黑客语言，像 1970 年代经典 Lisp：简洁、简单、可随意改造；还要有强大库，做黑客现在想做的事。

库这方面，我认为有机会在 Perl、Python 擅长的领域击败它们。未来几年许多新应用将是[服务器应用](https://paulgraham.com/road.html)。新 Lisp 完全能有和 Perl 一样好的字符串库；若还有强大的服务器应用库，可能非常流行。真黑客不会对一个用几次库调用就能解难题的新工具嗤之以鼻。记住，黑客很懒。

在核心语言里为服务器应用提供支持，可能是更大赢面。比如对多用户程序的显式支持，或在类型标签层级上的数据所有权。

服务器应用也回答了新 Lisp 用来折腾什么。让 Lisp 成为更好的 Unix 脚本语言没坏处（也很难更糟）。但我认为有些领域更容易击败现有语言。也许最好学 Tcl：把 Lisp 和完整的服务器应用支持系统一起提供。Lisp 天然适合服务器应用：词法闭包在 UI 只是一串网页时提供子程序效果；s 表达式与 HTML 映射自然，宏擅长生成 HTML。需要更好的服务器应用工具，也需要新 Lisp；两者会很好地协同。

### 黑客梦中的语言

总结而言，试着描述黑客梦中的编程语言。梦中语言[优美](https://paulgraham.com/taste.html)、干净、简短。有启动极快的交互式顶层。用极少代码就能解决常见问题；你写的程序里，几乎全是应用特有代码，其余都替你做好了。

语法简到极点：永远不必多敲字符，甚至很少按 Shift 键。

用大抽象可以极快写出第一版；要优化时，有极好的性能分析器告诉你该聚焦哪里。内层循环可以快得惊人，必要时甚至写内联字节码。

有大量好例子可学；语言足够直观，几分钟看例子就能上手。很少翻手册；手册薄，警告和限定少。

语言核心小，库强大、高度正交，与核心一样精心设计。库彼此配合；语言里一切像精密相机零件般契合。没有废弃项，没有为兼容而保留的部分。所有库源码易得。与操作系统及其他语言写的应用交互简单。

语言分层构建：高级抽象透明地由低级抽象构成，需要时你可以触及底层。

除非绝对必要，没有任何东西对你隐藏。语言提供抽象只为省你的事，不是为告诉你该做什么。事实上，语言鼓励你成为设计的平等参与者：连语法都可以改，你写的任何东西尽可能与预定义内容地位相同。
