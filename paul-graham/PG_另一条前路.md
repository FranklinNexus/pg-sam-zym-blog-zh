# 另一条前路

> 原文：<https://paulgraham.com/road.html>
> 来源：[paulgraham.com](https://paulgraham.com/)
> 英文标题：The Other Road Ahead
> 年代：2001 年 9 月（本文说明为何下一代软件多半在服务器上运行、对程序员意味着什么、为何对初创是机会；源自 BBN 实验室演讲。）

### 开篇：Viaweb 与 ASP

1995 年夏天，我和朋友 Robert Morris 决定创办一家初创公司。当时网景上市前的公关攻势正酣，媒体也在大谈在线商务。网上大概只有三十家真正的商店，全是手工做的。若在线商店会大量出现，就需要做商店的软件——我们决定来写。

头一周左右，我们打算做普通的桌面应用。后来有一天想到：让软件跑在我们自己的 Web 服务器上，用浏览器当界面。试着按这个方向重写，立刻清楚这才是正路。软件跑在服务器上，对用户和我们都会容易得多。

计划对了。如今这套软件以 Yahoo Store 之名，是最受欢迎的在线商店构建工具，用户约一万四千家。

创办 Viaweb 时，我们说软件跑在服务器上，几乎没人听得懂。直到一年后 Hotmail 上线，人们才开始明白。现在大家公认这是可行路线。我们当时做的事有了名字：应用程序服务提供商，即 ASP。

我认为下一代软件里，会有很大一块按这个模型来写。就连损失最大的 Microsoft，似乎也看出有些东西迟早要从桌面移走。若软件从桌面迁到服务器，开发者面对的世界会截然不同。本文描述的是我们——这个新世界的早期访客——所见的种种意外。就软件确实会迁上服务器而言，这里写的便是未来。

### 下一步是什么

回望桌面软件时代，我想我们会惊叹人们曾忍受的不便，就像如今惊叹早期车主的遭遇。头二三十年，你得是汽车行家才配拥有一辆车；但汽车赢太大了，许多非行家也想要。

计算机正处在类似阶段。你有一台台式机，最终会学到比你想知道的更多「里面发生了什么」。可美国过半家庭仍有一台。我母亲用电脑收发邮件、记账。大约一年前，她收到 Apple 来信，说可以折扣升级操作系统——一位六十五岁、只想用电脑处理邮件和账务的人，居然要考虑装新系统，这本身就不对。普通用户不该知道「操作系统」这个词，更不该知道「设备驱动」或「补丁」。

现在有另一种交付软件的方式，能把用户从系统管理员的角色里解放出来。基于 Web 的应用程序跑在 Web 服务器上，用网页当用户界面。对普通用户而言，这类软件通常比桌面软件更易用、更便宜、更便携、更可靠，且往往更强大。

基于 Web 的软件下，多数用户只需关心自己在用的应用。所有杂乱、易变的东西都在某处服务器上，由擅长此事的人维护。因此你通常不必为了用软件而专门拥有一台「计算机」——有键盘、屏幕和浏览器即可。也许还有无线联网，也许那就是你的手机。无论如何，它都是消费电子产品：大约两百美元，人们多半按外壳好不好看来选。你为互联网服务付的钱会超过硬件，就像现在为电话付的钱超过话机一样。[1]

一次点击往返服务器大约十分之一秒，Photoshop 这类高度交互的软件，用户仍希望计算在本地完成。但若看大多数人用电脑做什么，这点延迟不成问题。我母亲并不真需要台式机，像她这样的人还有很多。

### 用户的胜利

我家附近有辆车，保险杠贴纸写着「宁死不受不便」。多数人、多数时候，都会选最省力的那条路。若基于 Web 的软件胜出，多半因为它更方便——对用户和开发者，看起来都会如此。

纯 Web 应用只需联网的浏览器。因此随时随地可用。桌面软件装在一台机器上，就只能在那台机器上用；更糟的是，文件也被困在那台机器上。人们习惯网络之后，这种不便越来越刺眼。

楔子最薄的一端是基于 Web 的邮件。数百万人已明白：无论身在何处，都应能读邮件。能看邮件，为何不能看日历？能和同事讨论文档，为何不能一起改？数据何必困在远处某张桌子上的某台电脑里？

「你的计算机」这一概念正在消失，代之以「你的数据」。你应能从任何机器取回数据——更准确说，从任何客户端；客户端不必是计算机。客户端不应存数据，应像电话一样；事实上，客户端可能变成电话，或电话变成客户端。客户端越小，你越不该把数据放在上面：随身之物会丢、会被偷。PDA 落在出租车里，像磁盘崩溃，只是数据交到别人手里，而非蒸发。

纯 Web 软件下，数据和程序都不在客户端。无需安装即可使用；没有安装，就不必担心安装失败。应用与操作系统不可能不兼容，因为软件根本不在你的操作系统上跑。

无需安装，因此在「购买」前试用基于 Web 的软件会很容易、很普遍。任何 Web 应用都应能免费试驾——访问提供该应用的网站即可。Viaweb 整站就像一支大箭头，把用户指向试驾。

试完演示，注册服务理想情况下只需填一张短表（越短越好）。那应是用户要做的最后一件苦差。基于 Web 的软件，你应能免费获得新版本，无需额外付费、额外操作，甚至可能不知不觉。

升级不会像现在这样惊天动地。应用会悄悄变强。开发者得为此费些心思：设计得能在不迷惑用户的前提下更新。这是新问题，但有办法解决。

基于 Web 的应用，人人用同一版本，发现 bug 即可修复。因此 bug 应远少于桌面软件。Viaweb 时期，我怀疑我们同时存在的已知 bug 从未超过十个——比桌面软件好几个数量级。

Web 应用可同时被多人使用。协作类应用是显而易见的赢家；我赌一旦用户意识到可能，多数应用他们都会想要这点。例如两人同时编辑同一文档往往很有用。Viaweb 允许多人同时编辑站点，更多是因为这样写软件才对，而非预期用户需要——结果很多人确实这样用。

用 Web 应用，数据更安全。磁盘崩溃不会消失，但用户不会再听到。崩溃发生在服务器机房里。提供 Web 应用的公司会真做备份——不仅因为有专业系统管理员操心，更因为丢用户数据的 ASP 会惹上大麻烦。用户自己丢数据，只能生自己的气；公司替他们丢了，他们会怒得多。

最后，Web 应用应更不易受病毒侵害。客户端除浏览器外不跑别的，感染机会更小，本地也无数据可毁。攻击服务器本身的程序会发现防守严密。[2]

对用户而言，Web 应用压力更小。我想若窥视普通 Windows 用户内心，会发现对「符合上述描述」的软件有巨大、几乎未开发的渴望；释放出来会是强大力量。

### 代码之城

对开发者而言，Web 软件与桌面软件最显眼的区别是：Web 应用不是单块代码，而是多种程序的集合，而非一个大二进制。设计 Web 软件像设计一座城，而非一栋楼：除建筑外，还需要道路、路牌、公用设施、警消防，以及应对增长与各类灾难的规划。

Viaweb 的软件包括：用户直接交互的较大应用、这些程序调用的程序、后台常驻找问题的程序、出故障时尝试重启的程序、偶尔跑统计或建搜索索引的程序、我们显式运行以回收资源或迁移/恢复数据的程序、假扮用户测性能或暴露 bug 的程序、诊断网络故障的程序、做备份的程序、对接外部服务的接口、驱动一排实时服务器统计表盘的软件（访客觉得酷，对我们也不可或缺）、对开源软件的修改（含 bug 修复），以及大量配置文件和设置。被 Yahoo 收购后，Trevor Blackwell 写了一个精彩程序，在全国把商店迁到新服务器而不停机。程序给我们寻呼、给用户发传真和邮件、与信用卡处理器交易，并通过 socket、管道、HTTP 请求、ssh、UDP 包、共享内存和文件彼此通信。Viaweb 甚至有一部分是「没有程序」——Unix 安全要诀之一是不跑可能被用来闯入服务器的不必要工具。

不止软件。我们花大量时间想服务器配置。自己用组件攒服务器——部分是为了省钱，部分是为了拿到我们想要的配置。得考虑上游 ISP 到各骨干网的连接够不够快。我们一家一家地「考察」RAID 供应商。

但硬件不只是要担心的事。你控制硬件，就能为用户做更多。桌面应用可以规定最低硬件，却无法追加硬件。你若管服务器，一步就能给所有用户开通寻呼、发传真、电话下指令、处理信用卡等——装上相应硬件即可。我们总在找用硬件加功能的新法子，不仅用户高兴，也是与竞争对手区分的办法——他们（或因卖桌面软件，或因经 ISP 转售 Web 应用）无法直接控制硬件。

Web 应用里的软件是程序集合而非单一二进制，因此可用任意多种语言写。写桌面软件时，你实际上被迫用与底层操作系统相同的语言——即 C 和 C++。于是这些语言（尤其在经理和 VC 等非技术人员眼里）成了「严肃」开发的语言。但那只是桌面交付方式的产物。服务器软件可用任何语言。[3]

如今许多顶尖黑客用的是离 C/C++ 很远的语言：Perl、Python，甚至 Lisp。服务器软件下，没人能规定你用什么语言——你控整个系统，直到硬件。不同语言适合不同任务；每种情况选最合适的。有竞争对手时，「可以」即「必须」（稍后详述）——你若不用这优势，对手会用。

我们多数竞争对手用 C 和 C++，软件明显较差——部分因为绕不过 CGI 脚本的无状态：要改什么，所有改动得在一页完成，底部一个「更新」按钮。如我在别处写过，用 Lisp（许多人仍当是研究语言），我们让 Viaweb 编辑器更像桌面软件。

### 发布

这个新世界里最重要的变化之一是发布方式。桌面软件业，发布是巨大创伤：全公司汗流浃背，推出一块巨型代码。过程和成品都有显而易见的类比。

服务器软件下，你几乎可以像给自己写程序那样改。以一系列增量发布，而非偶尔大爆炸。典型桌面公司一年也许发一两个版本；Viaweb 常常一天发三到五个。

切到新模型后，你会意识到发布方式多大程度塑造了开发。桌面业许多最糟的问题，源于发布的灾难性。

一年只发一个新版本，你往往批量处理 bug。发布日前某时，你组装新版本，一半代码被撕掉重写，引入无数 bug。然后测试小组开始数，程序员按清单修。他们通常修不完，而且没人知道尽头在哪。像从池塘里捞碎石——你永远不太清楚软件内部发生了什么；至多得到统计意义上的正确。

服务器软件下，多数变化小且增量，本身就不易引入 bug。你也知道发布前最该测什么：最后改的那块。你对代码的把握会牢固得多。一般规则下，你知道里面在发生什么——源码不必背下来，但读源码像飞行员扫仪表，而非侦探解谜。

桌面软件滋生对 bug 的某种宿命论：你知道出货的东西 bug 成堆，还建了补偿机制（如补丁版）。何必再为多几个担心？很快你会发布明知有问题的整项功能。Apple 今年初就这么干：迫于发布新 OS 的压力（发布日已延四次），部分软件（CD/DVD 支持）未就绪。解法？先发布没有未完成部分的 OS，用户稍后自己装。

Web 软件下，你不必在能跑之前发布，能跑就可以立刻发布。

行业老手也许想：永不在能跑前发布听起来不错，但若你承诺某日前交付新版本呢？Web 软件下你不会这样承诺——因为没有版本。软件逐渐、持续地变；有些变化更大，但「版本」这个概念天然套不上 Web 软件。

若有人还记得 Viaweb，这可能听起来怪——我们总在宣布新版本。那完全是为公关。我们学到，商业媒体按版本号思考：主版本（第一位数字变）给大篇幅，点版本（小数点后新数字）一般一段。

有些竞争对手卖桌面软件，真有版本号；对我们而言，单这一事实就证明他们落后，却还能换来各种宣传。我们不想错过，也开始给软件标版本号。想要宣传时，我们列出自上次「发布」以来的所有功能，贴新版本号，发新闻稿说新版本立即可用。神奇的是，没人拆穿我们。

被收购时我们做了三次，所以在第 4 版——若没记错，4.1 版。Viaweb 变成 Yahoo Store 后，不再那么迫切地需要宣传，软件继续演进，版本号整套概念悄悄弃了。

### 错误

Web 软件另一大技术优势是：多数 bug 可重现。用户数据就在你磁盘上。有人搞坏软件，不必像桌面那样猜发生了什么——应能在他们打电话时重现。若应用内置了错误通知，你甚至可能已经知道。

Web 软件昼夜运行，所做一切立刻经受考验。Bug 很快露头。

软件公司有时被指责让用户帮着找 bug。我主张的正是这个。对 Web 软件这是好计划：bug 更少，且短暂。渐进发布，起始 bug 就少得多；能重现又能即时发布修复，多数 bug 一出现就能找到并修掉。我们从未多到需要正式 bug 跟踪系统。

当然发布前要测，重大 bug 不应上线。难免漏掉的少数会涉及边缘情况，只影响在有人投诉前撞上的少数用户。只要立刻修，对普通用户净效果是 bug 少得多。我怀疑普通 Viaweb 用户从未见过 bug。

修新 bug 比修旧 bug 容易。刚写的代码里找 bug 通常很快；一出现，往往在读源码前就知道哪儿错了——你潜意识里已在担心。修六个月前写的 bug（若一年一版，这是平均情况）工作量大得多；且因理解不够，更可能修得丑，甚至引入更多 bug。[4]

早抓 bug，复合 bug 也更少。复合 bug 是两个独立 bug 相互作用：下楼绊了一下，抓扶手时扶手脱手。软件里这类 bug 最难找，后果往往最严重。[5] 传统「先全弄坏再筛 bug」的做法天然产生大量复合 bug；一系列小变化发布的软件天然倾向不会——地板不断被扫干净，免得后来卡进什么。

用函数式编程会有帮助：避免副作用。商业软件里少见，研究论文里多见，对 Web 应用却非常有用。整程序写成纯函数很难，但大块可以；这些部分更易测——无状态——在不停做小改、小测时非常方便。Viaweb 编辑器大部分按此风格写；脚本语言 RTML 也做成纯函数式语言。

桌面业的人会难以相信：Viaweb 里 bug 几乎成了游戏。多数已发布 bug 涉及边缘情况，撞上的往往是把功能用到极限的高级用户。高级用户对 bug 更宽容，尤其 bug 可能是在加他们要求的功能时引入的。事实上 bug 稀少，且要做很复杂的事才见，高级用户常以抓到 bug 为荣——他们打电话给支持，精神更像得意而非愤怒，仿佛从我们这儿得分。

### 支持

能重现错误，会改变客户支持的做法。多数软件公司，支持是为让客户感觉好：要么问已知 bug，要么他们操作错了，你得猜原因。两种情况下你都学不到多少，于是把支持当麻烦事，尽量与开发隔离。

Viaweb 不是这样。支持免费——因为我们想听客户声音。有人出问题，我们想立刻知道，以便重现并发布修复。

因此 Viaweb 开发与支持始终紧挨着。支持人员离程序员约三十英尺，知道真 bug 的报告随时可打断任何事。我们会离开董事会去修严重 bug。

我们的支持方式让各方更高兴。客户很高兴——想象打支持热线却被当作带来重要消息的人。支持人员高兴，因为能真帮用户，而非念脚本。程序员高兴，因为能重现 bug，而非听含糊的二手报告。

即时修 bug 的政策改变了支持与黑客的关系。多数公司，支持是低薪人肉盾牌，黑客是创造世界的小上帝。报告 bug 的流程多半是单向：支持填表，经（可能）测试部门传给程序员，上待办清单。Viaweb 非常不同：客户报 bug 后一分钟内，支持可能站在程序员旁边，听他说「靠，你说得对，是 bug。」支持听到黑客说「你说得对」会特别高兴；他们以前带 bug 来，神情像猫叼来刚杀的老鼠。这也让他们判断 bug 严重性更谨慎——荣誉攸关。

被 Yahoo 收购后，支持迁离程序员。那时我们才意识到：他们实际上兼做测试，某种程度上也是营销。除抓 bug，他们还保管更模糊、类似 bug 的知识——比如迷惑用户的功能。[6] 他们也是代理焦点小组；问他们用户更想要哪两个新功能，他们总说得对。

### 士气

能立刻发布软件，是很大的动力。我常步行上班时想到要改什么，当天就改。大功能也如此。即便某事要写两周（很少项目更长），我知道做完就能在软件里看到效果。

若得等一年才发下一版，我会把多数想法搁置至少一阵。但想法会引出更多想法。你写作时是否注意到，成稿里一半想法是写的过程中想到的？软件亦然。实现一个想法会带来更多想法。搁置一个想法，不只延迟实现它，还损失实现它会引出的所有想法。事实上搁置可能抑制新想法：你开始想新功能，瞥见架子，心想「下一版要做的新事已经够多了」。

大公司不做功能，做计划。Viaweb 有时在这上面栽跟头。投资者和分析师问未来计划。诚实的答案是：没有计划。对想改进的事有大致想法，若知道怎么做早就做了。接下来六个月做什么？看起来收益最大的事。我不确定是否敢这样答，但那是真相。计划只是架子上想法的另一种说法。想到好主意，我们就实现。

Viaweb 与许多软件公司一样，多数代码有明确负责人。但负责就是真负责：除负责人外无人须批准（甚至知晓）发布。防止搞砸的保护只有怕在同伴面前出丑——那已足够。我也许给人印象是我们轻率地猛写代码。我们确实快，但在把软件放到那些服务器上之前，想得非常仔细。专注比慢更重要——海军飞行员因专注，能在夜间以 140 英里时速把四万磅飞机降在起伏的航母甲板上，比一般青少年切百吉饼还安全。

这种写法是双刃剑：对少数优秀、可信程序员的小团队效果好得多；对大公司里平庸者，坏主意由委员会而非提出者自己拦住。

### 布鲁克斯定律的逆转

幸运的是，Web 软件确实需要更少程序员。我曾在一家中型桌面软件公司工作，工程部一百多人，产品开发只有 13 人；其余搞发布、移植等。Web 软件至多只需这 13 人——没有发布、移植等。

Viaweb 只有三个人写的。[7] 我总面临加人压力——我们想被收购，买家很难为只有三个程序员的公司付高价。（解法：我们雇了更多人，但给他们开新项目。）

能用更少程序员写软件，省的不只是钱。Fred Brooks 在《人月神话》里指出：加人往往拖慢项目。开发者之间可能联系数随团队规模指数增长；团队越大，开会协商软件如何协同的时间越多，未预见的交互也会带来更多 bug。幸运的是，过程也可逆转：团队越小，开发效率指数级提高。我不记得 Viaweb 程序员开过真正的会；任何时刻要说的，走路去午饭的路上都说不完。

若这里有代价，那就是所有程序员都必须在某种程度上当系统管理员。托管软件时，得有人盯服务器；实践中能妥善做好的只有写软件的人。Viaweb 系统组件多、变化频，软件与基础设施之间没有明确边界——武断划界会束缚设计。因此我们一直希望某天（「几个月内」）一切稳定到能雇专人只管服务器，却从未发生。

我认为只要还在积极开发产品，不可能有别的方式。Web 软件绝不是你写好、提交、回家就完事——它是活物，此刻就在你的服务器上跑。严重 bug 可能不只搞垮一个用户进程，而是全部；代码 bug 若损坏磁盘数据，你得修。等等。我们发现不必每分钟盯服务器（第一年后），但务必盯紧最近改过的部分。你不会深夜发布代码然后回家。

### 观察用户

服务器软件下，你与代码贴得更近，也能与用户贴得更近。Intuit 以在零售店自我介绍、请求跟顾客回家而闻名。若你见过有人第一次用你的软件，你知道等待他们的会是多少意外。

软件应做用户以为它会做的事。但相信我，在观察用户之前，你无法知道用户会怎么想。服务器软件给你前所未有的用户行为信息——不限于小型人工焦点小组；你能看到每个用户的每次点击。得仔细想看什么，因不想侵犯隐私；但最一般的统计抽样也极有用。

用户在你服务器上时，不必依赖基准测试——基准测试是模拟用户；服务器软件下你能看真实用户。要优化什么，登录服务器看谁在吃 CPU。你也知道何时该停：我们最终让 Viaweb 编辑器受内存限制而非 CPU 限制；用户数据大小没什么简便办法再减，我们知道可以停了。

效率对服务器软件重要——你在付硬件钱。每服务器能支撑的用户数是资本成本的除数；软件非常高效，就能低价压倒对手仍盈利。Viaweb 我们把每用户资本成本压到约 5 美元；现在会更少，也许比寄首月账单还便宜。软件只要够高效，硬件就算免费了。

观察用户也能指导设计。Viaweb 有脚本语言 RTML，让高级用户自定义页面样式。我们发现 RTML 成了建议箱——用户只在预定义样式做不到时才用。例如编辑器最初把按钮栏横贯页面；不少用户用 RTML 把按钮放左侧后，我们在预定义样式里加了该选项（实为默认）。

最后，观察用户常能判断他们何时遇到麻烦。客户永远是对的——那是你需要修复的信号。Viaweb 获用户的关键是在线试驾：不是营销做的幻灯片，用户真用软件；约五分钟，他们建成一家真实、可运作的商店。试驾是我们获几乎所有新用户的方式；多数 Web 应用我想也会一样。用户顺利完成试驾就会喜欢产品；困惑或无聊就不会。因此任何能提高试驾完成率的事都会提高增长率。

我研究试驾者的点击轨迹，发现某步他们会困惑而点浏览器「后退」。（若你写 Web 应用，会发现「后退」是最有趣的哲学问题之一。）于是我在那步加消息：快完成了，提醒别点后退。Web 软件另一好处是改完即时反馈：完成试驾比例立刻从 60% 升到 90%。新用户数是完成试驾数的函数，仅这一改收入就增 50%。

### 金钱

九十年代初，我读到一篇文章说，软件是一门订阅生意。起初这说法显得很愤世嫉俗。后来我才明白，它反映的是现实：软件开发是一个持续进行的过程。我认为，与其强迫用户不断购买、安装新版本才能继续付钱，不如公开收取订阅费来得干脆。幸运的是，订阅正是 Web 应用最自然的计费方式。

托管应用这一环节，大概会由公司来承担，不太可能由免费软件填补。托管应用压力很大，也有实实在在的开销，没人会愿意免费干这份活。

对公司来说，基于 Web 的应用是理想的收入来源。每个季度不是从零起步，而是有持续的收入流。软件逐渐演进，不必担心某个新模型会彻底失败——根本上不必推出什么「新型号」；如果你做了用户讨厌的事，马上就会知道。也没有收不回的账款——对方不付钱，你就关掉服务。同样不存在盗版问题。

最后这条「优势」说不定会变成麻烦。一定程度的盗版对软件公司其实有好处：若某个用户本来绝不可能以任何价格购买你的软件，他用盗版版，你并没有损失。事实上你还赚了——他多了一个用户，帮你把软件做成行业标准，或者日后——比如高中毕业后——可能买一份正版。

公司能时会做差别定价，意思是向每位客户收取他承受得起的最高价。[8] 软件特别适合差别定价，因为边际成本接近零。这就是为什么有些软件在 Sun 上运行比在 Intel 上贵：用 Sun 的公司并不打算省那点钱，可以放心收更高价。盗版实际上是最低档的差别定价。我认为软件公司明白这一点，并故意对某些盗版睁一只眼闭一只眼。[9] 但在服务器软件模式下，他们得另找办法。

基于 Web 的软件卖得好，尤其相对桌面软件而言，因为它买起来容易。你可能以为人们是先决定要买某样东西，然后再去买，这是两个独立步骤——创办 Viaweb 之前，我也是这么模糊地想的。事实上第二步会反噬第一步：如果某样东西很难买，人们就会改变主意，到底要不要它；反过来，越容易买，卖得就越多。有了 Amazon，我会买更多书。基于 Web 的软件大概是最容易购买的东西，尤其刚刚在线体验过演示之后。用户最好只需输入信用卡号（要他们做更多，后果自负）。

有时 Web 应用通过 ISP 充当经销商来提供。这是馊主意。你必须自己管理服务器，因为需要不断改进硬件和软件。一旦放弃对服务器的直接控制，就放弃了开发 Web 应用的大部分优势。我们有好几个竞争对手就是这样自断后路——通常，我想，是被西装革履的人冲昏了头脑：他们被这条潜在的巨大渠道迷住了，却没意识到它会毁掉他们希望通过这条渠道销售的产品。通过 ISP 卖 Web 应用，就像通过自动售货机卖寿司。

### 客户

客户会是谁？在 Viaweb，最初是个人和小公司。我认为 Web 应用多半也会如此——这些用户更愿意尝试新事物，部分因为更灵活，部分因为他们需要新技术带来的更低成本。

基于 Web 的应用往往也是大公司的最佳选择（尽管他们会慢慢才意识到）。最好的企业内部网络就是互联网本身。如果一家公司使用的是真正的 Web 应用，软件会运行得更好，服务器管理会更专业，员工无论身在何处都能访问系统。

反对这种做法的论点通常集中在安全上：员工访问方便了，坏人也方便。有些大商家不愿意用 Viaweb，因为觉得客户的信用卡信息放在自己的服务器上更安全。要外交辞令地说这事儿很难，但事实上，数据在我们手里几乎肯定比在他们手里更安全：谁更有能力雇到好人才来管安全——一家整个生意就是跑服务器的科技初创，还是一家服装零售商？我们不仅人更行，而且更操心。如果有人攻破服装零售商的服务器，最多影响一家商户，大概还能压下消息，最坏也不过炒掉一个人。如果有人攻破我们的系统，可能影响数千家商户，大概会上 CNet 新闻，可能让我们破产。

如果想保资金安全，你会把钱塞在床垫下，还是存进银行？这个论证适用于服务器管理的方方面面：不只是安全，还有正常运行时间、带宽、负载管理、备份等等。我们的生死取决于把这些事做对。服务器出问题对我们是大忌——就像危险玩具对玩具制造商，或沙门氏菌疫情对食品加工厂。

采用 Web 应用的大公司，在某种程度上是在把信息技术外包出去。听起来很激进，但我认为这总体上是好主意。公司很可能因此得到比内部系统管理员更好的服务。系统管理员可能会变得暴躁、反应迟钝，因为他们不直接面对竞争压力：销售人员要对付客户，开发人员要对付竞争对手的软件，系统管理员却像老光棍一样，很少有外力约束他们。[10] 在 Viaweb，我们有大把外力约束：打电话来的是客户，不是同事。服务器一旦卡死，我们立刻跳起来——多年后再想，仍会肾上腺素直飙。

因此，Web 应用通常也是大公司的正确答案。不过他们会最后才意识到这一点，就像当年对台式机一样——部分也因为同一个理由：说服大公司相信他们需要更昂贵的东西，值得花大笔钱去做。

富裕的客户总有买昂贵方案的倾向，哪怕便宜方案更好——因为兜售昂贵方案的人可以花更多钱来卖。在 Viaweb，我们常碰到这种情况。我们输掉过几个高端商户，因为他们被网络咨询公司说服：花五十万美元在自己服务器上建一套定制在线商店更划算。他们通常并非如此——不止一家在圣诞购物季负载上升时才发现这一点。Viaweb 比大多数商户最终拿到的系统复杂得多，但我们每个月只收三百美元，请不起一支衣冠楚楚、说话权威的队伍去做现场演示，没法把这一点告诉他们。

大公司多付的很大一部分钱，是卖昂贵东西给他们的成本。（如果国防部花一千美元买马桶座，部分原因是把马桶座卖到一千美元本身就代价不菲。）这也是内网软件会继续繁荣的原因之一，尽管那大概不是好主意——只是更贵。对这个两难局面你无能为力；最好的策略是先瞄准小客户，其余的会慢慢跟上来。

### 服务器之子

在服务器上运行软件并不新鲜——事实上，那是旧模式：大型机上的应用全是服务器端的。如果服务器端软件这么好，上次为什么输？为什么台式机盖过了大型机？

起初台式机看起来不像多大威胁。第一批用户全是黑客——当时叫爱好者。他们喜欢微型计算机，因为便宜——第一次可以拥有自己的计算机。「个人计算机」这个词如今已是日常用语，但刚出现时故意说得很大胆，就像今天说「个人卫星」一样。

台式机为什么占上风？我认为是因为软件更好。而微型机软件之所以更好，是因为小公司能写。

我想很多人没意识到，初创公司在最早阶段是多么脆弱、多么摇摆不定。许多初创几乎是偶然起步——两个人，有日职或还在上学，写一个原型，若有希望也许能变成公司。在这个幼虫阶段，任何重大障碍都会让初创当场死掉。写大型机软件需要的前置投入太大：开发机很贵，客户是大公司，你需要一支看起来像回事的销售队伍。为大型机软件创办初创，比晚上在自己的 Apple II 上随手捣鼓严肃得多。因此没有多少初创会去写大型机应用。

台式机的到来激发了大量新软件，因为为它们写应用对幼虫期初创来说似乎够得着：开发成本低，客户是个人，可以通过电脑店甚至邮购触达。

把台式机推入主流的应用是 VisiCalc，第一款电子表格。两个家伙在阁楼里写的，却能做大型机软件做不到的事。[11] VisiCalc 在当时是一项巨大进步，人们专门买 Apple II 来跑它。趋势由此开始：台式机赢，是因为初创公司为它们写软件。

这一次，服务器端软件看起来会成功，因为初创会写它。计算机现在非常便宜，你可以像我们一样，用一台台式机当服务器起步。廉价处理器已经吞掉了工作站市场（这个词现在很少听到了），并且已经吃掉了大部分服务器市场；Yahoo 的服务器处理着互联网上最高的负载之一，用的全是和你桌上一样的廉价 Intel 处理器。写完软件，要卖它你只需要一个网站。我们几乎所有用户都是通过口碑和媒体报道，直接来到我们的网站。[12]

Viaweb 是典型处于幼虫期的初创。我们害怕开办公司，头几个月把整个事情当作一项随时可能取消的实验来安慰自己，让自己好过些。幸好除了技术问题，几乎没有别的障碍。写软件那阵子，我们的 Web 服务器就是开发用的那台台式机，用拨号线连到外界。那阶段我们的开支只有食宿。

如今更有理由让初创写基于 Web 的软件，因为写桌面软件已经不那么有趣了。如果你想写桌面软件，就得按 Microsoft 的规矩来，调用他们的 API，绕过他们充满 bug 的操作系统。如果你写出的东西火起来了，你也许会发现自己不过是在为 Microsoft 做市场调研。

如果一家公司想做一块平台，让初创能在上面搭建，就必须做成黑客们自己也想用的东西：便宜、设计好。Mac 刚出来时黑客很喜欢，很多人为它写软件。[13] Windows 上就少见多了——因为黑客不用 Windows。擅长写软件的人现在多半跑 Linux 或 FreeBSD。

我不认为我们会为写桌面软件而创办初创——桌面软件得在 Windows 上跑，而在写 Windows 软件之前，你得先使用 Windows。Web 让我们绕道 Windows，通过浏览器把跑在 Unix 上的软件直接交给用户。这是一种解放的前景，很像二十五年前个人电脑到来时的感觉。

### 微软

台式机刚出现时，IBM 是人人害怕的巨人——现在很难想象，但我记得那种感觉很清楚。如今令人畏惧的巨人是 Microsoft；我不认为他们像 IBM 那样对威胁视而不见——Microsoft 正是刻意在 IBM 的盲区内建立了自己的生意。

我前面说过，我母亲并不真的需要台式机；大多数用户大概也不需要。这对 Microsoft 是个问题，他们清楚。如果应用程序跑在远程服务器上，没人需要 Windows。Microsoft 会怎么办？他们能不能用对桌面的控制来阻止或限制这一代新软件？

我猜 Microsoft 会搞某种服务器/桌面混合方案，让操作系统与他们控制的服务器协同工作。至少，文件会为需要的用户集中存储、随时可用。我不指望 Microsoft 会走到极端——所有计算都在服务器上完成，客户端只有一个浏览器——只要还能避免。如果你只需要浏览器当客户端，客户端上就不需要 Microsoft；如果 Microsoft 控制不住客户端，就没法把用户引向他们的服务器端应用。

我认为 Microsoft 很难把精灵关回瓶子里：客户端类型太多，他们控不过来。如果 Microsoft 的应用只能在某些客户端上工作，竞争对手就能靠提供任何客户端都能用的应用来赢过他们。[14]

在 Web 应用的世界里，Microsoft 没有自动占坑的位置。他们也许能给自己辟出一块地盘，但我不认为他们会像统治桌面应用世界那样统治这个新领域。

与其说是竞争对手把他们绊倒，不如说他们会绊倒自己。随着 Web 软件的兴起，他们面对的不只是技术问题，还有自己的一厢情愿。他们需要蚕食现有业务——我看不出他们会面对这一点。当年把他们带到今天的这种专一，现在会反噬他们。IBM 当年处境一模一样，也没能驾驭局面：IBM 晚且半心半意地进入微型机市场，因为他们对威胁现金牛——大型机计算——心存矛盾。Microsoft 同样会因为想保住桌面而步履维艰。现金牛可以是背上一个死沉的包袱。

我不是说没人会统治服务器端应用——迟早可能有人会。但我认为会有一段相当长、相当欢快的混乱期，就像微型机早期那样。那是初创的好时光；许多小公司靠做酷东西而繁荣。

### 初创，且更甚

经典的初创公司快、非正式，人少钱少。这几个人非常努力地工作，技术会放大他们决策的效果。如果赢了，就赢得很大。

在写 Web 应用的初创里，一切与初创相关的东西都被推到极致。你可以用更少的人、更少的钱写产品并上线。你必须更快，可以更加非正式。字面上，你可以三个家伙坐在公寓客厅、在 ISP 托管一台服务器就上线——我们就是这么干的。

随着时间推移，团队越来越小、越来越快、越来越随便。1960 年，软件开发意味着一屋子戴角质框眼镜、系窄黑领带的男人，在 IBM 编码表格上一天写十行代码。1980 年是八到十个穿牛仔裤进办公室的人，在 vt100 终端上打字。现在是两个家伙坐在客厅用笔记本电脑。（原来牛仔裤也不是随便的最终形态。）

初创很有压力，不幸的是 Web 应用把这一点也推到极致。许多软件公司，尤其早期，都会有一段开发人员睡在办公桌下的时期。Web 软件令人不安之处在于，没有什么能阻止这种情况变成常态。那些睡办公桌下的故事通常以「终于发布了，大家回家睡一周」告终——Web 软件永远不会「发布」。你可以一天干十六小时，想干多久干多久；因为你能，竞争对手也能，于是往往会被迫跟上。你能，所以你必须——这是 Parkinson 定律的反向运行。

最糟的不是工时，而是责任。程序员和系统管理员传统上各有各的担心：程序员担心 bug，系统管理员担心基础设施。程序员也许整天埋在源码里，但总有某个时刻可以回家，把工作抛在脑后。系统管理员几乎离不开工作，但凌晨四点被寻呼机叫醒时，通常不用做很复杂的事。在 Web 应用里，这两种压力合在一起：程序员变成了系统管理员，却没有通常能让这份工作忍得住的那种清晰界限。

在 Viaweb，头六个月我们只写软件，干的是早期初创惯常的长工时。在桌面软件公司，这本该是辛苦工作那段，但和下一阶段——把用户接到我们的服务器上——相比，感觉像度假。把 Viaweb 卖给 Yahoo，第二大好处（仅次于钱）是能把整件事的最终责任甩给一家大公司。

桌面软件迫使用户当系统管理员；Web 软件迫使程序员当。总压力更小，但程序员压力更大——对与大公司竞争的初创不一定是坏消息。[15] Web 应用提供了一条直截了当的路，让你比对手更拼。没有初创会嫌这个要求太高。

### 够用就好

可能打消你写 Web 应用念头的一件事，是 Web 页作为用户界面太蹩脚。我承认这是问题；我们确实希望给 HTML 和 HTTP 加几样东西。但重要的是：Web 页够用了。

这与首批微型机有相似之处。那些机器里的处理器，本来并不是要当计算机的 CPU，而是为交通灯之类的东西设计的。但 Ed Roberts 等人（设计了 Altair）意识到它们够用了——把其中一颗芯片配上一些内存（第一台 Altair 只有 256 字节）和前面板开关，就是一台能用的计算机。能拥有自己的计算机太令人兴奋了，尽管功能有限，仍有不少人想买。

Web 页本来也不是为应用界面设计的，但够用了。对相当数量的用户来说，任何浏览器都能用的软件，其好处本身就足以抵消界面上的任何别扭。也许你用 HTML 写不出最好看的电子表格，但能写出一个多人、在不同地点同时使用、无需专门客户端、可接入实时数据流、或在某些条件触发时给你发寻呼的表格。更重要的是，你能写出尚无名字的新型应用——VisiCalc 不只是大型机应用的微型机版，而是一种全新类型。

当然，服务器端应用不必是基于 Web 的——你可以有别的客户端。但我相当确定那是坏主意。假设人人都会安装你的客户端，这听起来很方便——方便到你会轻易说服自己他们真的会装——但如果他们不装，你就完了。基于 Web 的软件对客户端不做任何假设，Web 能工作的地方它就能工作——这已经是很大优势，随着新 Web 设备涌现，优势还会更大。用户会喜欢你，因为软件就是能用；你的生活也会更轻松，因为不必为每个新客户端微调。[16]

我觉得我和任何人一样密切注视 Web 的演进，却无法预测客户端会怎样。融合大概会来，但往哪融？我选不出赢家。有一件事我可以预测：AOL 与 Microsoft 的冲突。无论 Microsoft 的 .NET 最终是什么，它大概都会把桌面和服务器连起来。除非 AOL 反击，否则要么被挤到一边，要么变成 Microsoft 客户端与服务器软件之间的管道。如果 Microsoft 与 AOL 打客户端战争，两边都肯定能用的，只有浏览 Web——也就是说，Web 应用是唯一到处都能用的应用。

这一切会如何展开？我不知道。但如果你押注 Web 应用，不必知道——没人能破坏这一点而不破坏浏览本身。Web 也许不是交付软件的唯一方式，但它是现在能工作、并且还会长期工作的方式。Web 应用开发成本低，最小的初创也容易交付。工作很多，而且特别累人，但那只会让初创的胜算更好。

### 何不？

E. B. White 曾从一个农民朋友那里听到一件有趣的事：许多带电围栏其实根本没有电流。牛显然学会了远离它们，之后就不再需要电流了。「起来吧，牛们！」他写道，「趁独裁者酣睡，夺取你们的自由！」

如果你是个黑客，想过某天创办初创，大概有两件事拦着你：一是不懂商业，二是怕竞争。这两道栅栏都没有通电。

关于商业，你只需要懂两件事：做用户热爱的东西，赚的比花的多。这两点对了，你就领先于大多数初创；其余可以边走边学。起初你也许还赚不过花，只要差距在快速缩小就没事。起步资金少，至少会养成节俭的习惯——花得越少，越容易赚比花多。幸运的是，上线一个 Web 应用可以非常便宜：我们不到一万美元就上线了，今天会更便宜。我们得花几千买服务器、几千买 SSL（当时卖 SSL 的只有 Netscape）。现在租一台更强大的服务器，SSL 包含在内，比我们当年单独付的带宽还少；上线一个 Web 应用，现在可能还不如一把高档办公椅贵。

至于做用户热爱的东西，这里有些一般性建议。先做干净、简单、你自己想用的东西。快速推出 1.0，然后继续改进，仔细听用户意见。客户永远是对的，但不同客户在不同方面是对的：最不成熟的用户告诉你该简化和澄清什么，最老练的告诉你该加什么功能。软件最好做到易用，办法是把默认值设对，而不是限制用户选择。如果竞争对手的软件很蹩脚，别因此自满——比较的标准是你的软件本来可以做成什么样，而不是对手碰巧有什么。一直用自己的软件；Viaweb 本是在线商店构建器，我们也用它做自己的网站。别只因职务头衔就听营销、设计师或产品经理的——有好主意就用，但决定权在你；软件必须由懂设计的黑客来设计，而不是略懂软件的设计师。如果你设计软件的能力赶不上实现它的能力，就别开初创。

再说竞争。你怕的大概不是像你这样的黑客小团体，而是真正的公司——有办公室、商业计划、销售人员，对吧？其实，他们比你更怕你，而且他们是对的。几个黑客琢磨清楚怎么租办公室或雇销售，比任何规模的公司把软件写出来容易得多。我两边都待过，我知道。Viaweb 被 Yahoo 收购后，我突然在大公司工作，像试图在齐腰深的水里跑步。

我不是要贬低 Yahoo——他们有一些好黑客，高层管理也很能踢屁股；对大公司来说，这已经很难得了。但他们仍然只有小初创大约十分之一的产出。没有哪家大公司能好太多。Microsoft 可怕，在于一家公司这么大还能开发软件——像一座会走路的山。

别被吓住。你能做 Microsoft 做不到的事，正如他们能做你做不到的事。没人能阻止你。开发 Web 应用不必向任何人请示——不必签授权协议、不必争取零售货架、不必低声下气把应用捆绑进操作系统。你可以把软件直接送到浏览器里，没人能挡在你和潜在用户之间，除非阻止他们浏览 Web。

你也许不信，但我向你保证：Microsoft 怕你。自满的中层经理也许不怕，Bill 怕——因为他曾经就是你，在 1975 年，上一次出现新的软件交付方式的时候。
