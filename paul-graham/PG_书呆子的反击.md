# 书呆子的反击

> 原文：<https://paulgraham.com/icad.html>
> 来源：[paulgraham.com](https://paulgraham.com/)
> 英文标题：Revenge of the Nerds
> 年代：2002 年 5 月

### 尖头老板

软件业里，尖头学者与另一股同样难缠的力量——尖头老板——之间，斗争从未停息。谁不知道尖头老板？科技圈里大多数人不仅认得这个漫画人物，还知道公司里以他为原型的那位真人。

尖头老板奇迹般地同时具备两种各自常见、却极少同现的特质：（a）对技术一无所知；（b）对技术意见极重。比如你要写软件：他不懂软件该怎么工作，分不清编程语言 A 和 B，却笃定你该用哪种语言写——没错，Java。

### 语言等价谬误

他脑子里大致是这样：Java 是标准。媒体天天写，所以一定是。既然是标准，用了就不会挨骂。而且永远不缺 Java 程序员；为我干活的人若像往常那样莫名离职，我也好找人顶上。

听起来不算离谱——但全建立在一个未言明的假设上，而那假设是错的：**尖头老板相信所有编程语言都差不多等价。** 若果真如此，他当然该选「大家都在用的」。可语言并不等价；我甚至不必深入比较各语言差异就能说明这一点。

1992 年你若问尖头老板该用什么语言写软件，他会像今天一样不假思索：C++。若语言都等价，他的答案为何会变？Java 的开发者又何必造新语言？造新语言，显然是因为觉得比已有的更好；Gosling 在第一份 Java 白皮书里也明确说，Java 是为修补 C++ 的某些问题而设计的。所以：**语言并不等价。** 从尖头老板的脑回路经 Java 再沿历史回溯，你最终得到的结论与出发点矛盾。

谁对？Gosling，还是尖头老板？毫不意外，Gosling。某些问题上，某些语言就是更好——这引出一串有趣问题：Java 相对 C++ 更好，好在哪里？何时 Java 更好、何时 C++ 更好？有没有别的语言比两者都好？

### 两个联立方程

一旦认真想这些问题，你就捅了马蜂窝。若尖头老板必须面对问题的全部复杂度，大脑会炸。只要相信语言等价，他只需选「势头最旺」的那种——这更多是时尚而非技术，他或许也能蒙对。可若语言确有高下，他忽然要同时解两个方程：在二十来种主流语言里，哪种最适合眼前问题；每种语言找到程序员、库等的概率——两件事他一概不懂。

相信「语言等价」的坏处是：不真实。好处是：生活简单得多。我想这正是此说流传甚广的原因——它让人舒服。

### 酷语言的层级

我们知道 Java 一定不错，因为它是时髦的新语言——是吗？远观编程语言世界，Java 像是最新潮流（再远些，只见 Sun 砸钱的大广告牌）。近看则发现「酷」有等级：黑客亚文化里，Perl 比 Java 酷得多——Slashdot 就是 Perl 生成的；那些人不大会用 Java Server Pages。还有更新的 Python，用户常瞧不起 Perl，后面还有[更多在排队的语言](https://paulgraham.com/accgen.html)。

按 Java、Perl、Python 的顺序看，若你是 Lisp 黑客，会注意到有趣规律：**一种比一种更像 Lisp。** Python 甚至抄了 Lisp 黑客眼里算「错误」的特性；简单 Lisp 程序可逐行译成 Python。2002 年了，编程语言才差不多追上 1958 年。

### 赶上数学

我的意思是：Lisp 1958 年由 John McCarthy 提出，流行语言直到现在才赶上他当时的想法。怎么可能？计算机技术不是变很快吗？1958 年的计算机是冰箱大小的巨兽，算力不如手表[1]——这么老的东西怎会还有用，更别说优于新发展？

因为 Lisp 起初并非按今天我们说的「编程语言」来设计——我们说的编程语言，是用来告诉计算机做什么的。McCarthy 后来确实打算往这方向做，但我们得到的 Lisp，其实源于他的一项[理论练习](https://paulgraham.com/rootsoflisp.html)：给图灵机找一个更方便的替代定义。

### McCarthy 与 eval

McCarthy 后来说，要说明 Lisp 比图灵机更简洁，可以写一个通用 Lisp 函数，证明它比通用图灵机描述更短、更易懂——那就是 Lisp 函数 [eval](https://sep.turbifycdn.com/ty/cdn/paulgraham/jmc.lisp?t=1772732230&)，计算 Lisp 表达式的值。写 eval 需要发明一种把 Lisp 函数表示成 Lisp 数据的记号；为论文而设，并未想到会用来写实际程序。

1958 年末某时，McCarthy 的研究生 Steve Russell[2] 看了 eval 的定义，意识到译成机器码就是 Lisp 解释器——当时是大惊喜。McCarthy 后来回忆：

> Steve Russell 说，我来编这个 eval……我说，呵呵，你把理论和实践搞混了，这个 eval 是给人读的，不是拿来算的。但他还是做了：把我论文里的 eval 编译成 [IBM] 704 机器码，修 bug，然后宣传成 Lisp 解释器——它确实是。于是 Lisp 基本上就有了今天的形态……

几周内，理论练习变成了真正的编程语言——而且比他原想的更强大。

### 两棵进化树

1950 年代语言未过时的简短解释：Lisp 不是技术，是数学；数学不会过时。该拿来比的是快速排序（1960 年发现，至今仍是通用排序里最快的之一），不是 1950 年代硬件。

1950 年代还存活下来的另一语言是 Fortran，代表相反的设计路线：Lisp 是理论意外变成语言；Fortran 有意做成编程语言，但以今天的标准看极低级。[Fortran I](https://paulgraham.com/history.html)（1956）与当今 Fortran 几乎是两种动物——差不多是带数学的汇编；某些方面比后来的汇编还弱（没有子程序，只有分支）。现今 Fortran 可以说已更接近 Lisp，而非 Fortran I。

Lisp 与 Fortran 是两棵独立进化树的主干：一棵植根于数学，一棵植根于机器体系结构；此后两树一直在收敛。Lisp 起步强，二十年间变快；所谓主流语言起步快，四十年间渐强，如今最先进的已相当接近 Lisp——接近，仍缺几样东西。

### Lisp 的九条想法

Lisp 初现时体现了九条新想法。有些我们已视为理所当然，有些只在更高级语言里见到，两条仍为 Lisp 独有。按被主流采纳的顺序：

1. **条件分支**——if-then-else。Fortran I 没有，只有紧贴底层机器指令的条件跳转。
2. **函数类型**——函数与整数、字符串一样是数据类型，有字面量，可存变量、作参数。
3. **递归**——Lisp 是第一个支持的。
4. **动态类型**——变量实质都是指针；类型在值上，不在变量上；赋值/绑定是复制指针，而非复制所指内容。
5. **垃圾回收**。
6. **程序由表达式组成**——Lisp 程序是表达式树，每个节点返回值。Fortran 及多数后继语言区分表达式与语句：Fortran I 不能嵌套语句，数学需要表达式，别的没必要返回值——块结构语言出现后这限制没了，但区分已根深蒂固，从 Fortran 传到 Algol 再传到后代。
7. **符号类型**——符号实质是指向哈希表里字符串的指针；比较相等性只需比指针，不必逐字符比。
8. **用符号与常量树表示代码**。
9. **语言始终在场**——读时、编译时、运行时无真正分界：读时可编译/运行，编译时可读/运行，运行时可读/编译。读时运行代码可重编程 Lisp 语法；编译时运行是宏的基础；运行时编译是 Emacs 等扩展语言的基础；运行时读取使程序用 s 表达式通信——这想法最近又被发明成 XML。

Lisp 刚出现时，这些与受 1950 年代末硬件支配的普通实践相距甚远。默认语言（体现在一代代流行语言里）已逐渐向 Lisp 演化：1–5 已普及，6 开始进入主流，Python 有 7 的某种形式但似乎无对应语法。

### 无语法与宏

第 8 条或许最耐人寻味。8 和 9 进 Lisp 纯属偶然——Russell 实现了 McCarthy 从未打算实现的东西——却造就了 Lisp 的怪相与最鲜明的特征。Lisp 看起来怪，与其说语法怪，不如说**没有语法**：你直接在别的语言解析时在幕后建起的语法树里写程序，而树由列表构成——Lisp 的数据结构。

用语言自己的数据结构表达语言，是非常强大的特性。8 和 9 合起来意味着：**你可以写「写程序的程序」。** 在 Lisp 里这是家常便饭，最常见的方式是**宏**。

「宏」在 Lisp 里不是别的语言里的意思。Lisp 宏可以是缩写，也可以是新语言的编译器。要真懂 Lisp 或拓宽视野，值得[多学宏](https://paulgraham.com/onlisp.html)。据我所知，宏（Lisp 意义下）仍为 Lisp 独有——部分因为要有宏，语言可能就得像 Lisp 一样怪；部分因为若加上这最后一档能力，你不能再声称发明了新语言，只能说发明了 Lisp 新方言。多半当玩笑说，但千真万确：若语言有 car、cdr、cons、quote、cond、atom、eq 及用列表表示函数的记号，其余 Lisp 都能由此构建——这正是 Lisp 的决定性特质；McCarthy 为此才给 Lisp 现在的形状。

### 语言有多重要

若 Lisp 代表主流语言渐近逼近的某种极限——是否该用它写软件？用弱语言损失多少？有时不站在创新最前沿是否更明智？流行本身是否算理由？尖头老板想选「好雇程序员」的语言，错了吗？

当然有项目语言不太要紧。一般规律：**应用越苛刻，强语言的杠杆越大。** 但很多项目并不苛刻；多数编程是写小粘合程序——用你熟的语言、库够好就行。Windows 应用间传数据？Visual Basic 也行。Lisp 也能写小粘合程序（我当桌面计算器用）；但 Lisp 这类语言最大的收益在光谱另一端：要写复杂程序，在激烈竞争下解难题。

### ITA 的案例

好例子是 ITA Software 授权给 Orbitz 的[机票搜索程序](https://paulgraham.com/carl.html)。他们进入 Travelocity、Expedia 两巨头把持的市场，却在技术上把对手打得落花流水。ITA 核心是一个 20 万行 Common Lisp 程序，搜索空间比对手大几个数量级——对手显然仍用大型机时代手法（ITA 某种意义上也在用大型机时代的语言）。我没见过 ITA 代码，但据其顶尖黑客说用大量宏——并不意外。

### 向心力：互操作、库与招聘

冷门技术并非零成本；尖头老板担心并非全错——但他不懂风险，往往放大。

用冷门语言可能有三问题：与别种语言写的程序配合差；库少；难雇程序员。

第一点的权重取决于是否控制全系统。软件须跑在用户机器上、叠在有缺陷的闭源操作系统上（不点名），用操作系统同语或许有优势；若控制全系统且有全部源码（ITA 大概正是如此），用什么语言都行，不兼容自己修。

服务器应用可用最先进技术——Jonathan Erickson 所谓「[编程语言复兴](http://www.byte.com/documents/s=1821/byt20011214s0003/)」的主因。我们听说 Perl、Python，不是因为有人用它们写 Windows 应用，而是因为服务器上在用。软件[离开桌面、上服务器](https://paulgraham.com/road.html)（连微软似乎也已接受这一未来），用中庸技术的压力会减小。

库的重要性也看应用。问题不太苛刻时，库的便利可压过语言内在能力。盈亏点难精确说，但肯定达不到你会叫「应用」的门槛。公司若自认软件业、要写将成为产品的应用，多半要几个黑客、至少六个月——这种规模下，强语言大概开始压过现成库的便利。

第三点——难雇程序员——我认为是障眼法。要雇几个黑客？软件最好十人以下团队开发。任何「有人听说过」的语言，这规模都不难雇人。找不到十个 Lisp 黑客，公司大概选错了做软件的城市。

选更强语言大概缩小团队：（a）可能不需要那么多黑客；（b）用更高级语言的黑客往往更聪明。不是说不会有压力要用所谓「标准」技术。Viaweb（现 Yahoo Store）用 Lisp 让风投和潜在收购方侧目；我们也因用普通 Intel 当服务器而非 Sun 的「工业级」机器、用当时还鲜为人知的开源 Unix 变体 FreeBSD 而非 Windows NT、忽略现已无人记得的 [SET](http://news.com.com/2100-1017-225723.html) 电商「标准」等而引人侧目。

不能让穿西装的人替你做技术决策。用 Lisp 让部分收购方警觉？有一点——但若不用 Lisp，写不出让他们想买我们的软件。他们眼里的反常，其实是因果。做初创公司，别为取悦风投或收购方设计产品；为用户设计。赢得用户，其余自会跟上；赢不了，没人会在意你的技术选择有多令人安心的正统。

### 平庸的代价

用弱语言损失多少？其实有数据。能力最方便的度量大概是[代码量](https://paulgraham.com/power.html)：高级语言给更大抽象——更大砖块，砌同样高的墙要更少砖。语言越强，程序越短（不是字符数，而是独立元素数）。

### 自下而上编程

强语言如何让你写更短程序？若语言允许，可用[自下而上编程](https://paulgraham.com/progbot.html)：不在基语言里直接写应用，而在基语言上再建一门「写这类程序的语言」，再用它写程序。组合代码可远短于全用基语言写——多数压缩算法就这么工作。自下而上程序也易改：很多情况下语言层根本不用动。

代码量重要，因为写程序时间多半取决于长度。别种语言里三倍长，就三倍时间——雇更多人绕不过去；超过一定规模，新人反而拖后腿。Fred Brooks 在《人月神话》里描述过，我见闻皆印证。

Lisp 比 C 短多少？常见说法是 7–10 倍。[New Architect](http://www.newarchitectmag.com/documents/s=2286/new1015626014044/) 登 ITA 一文引其总裁说「一行 Lisp 抵二十行 C」——若属实可采信；ITA 软件里 C/C++ 与 Lisp 都有，是经验之谈。我猜倍数非恒定：问题越难、程序员越聪明，倍数越大；好黑客能从好工具榨出更多。

与 ITA 竞争若选 C，他们开发速度可能是你的二十倍：你花一年做新功能，他们不到三周可复制；他们三个月做新东西，你要五年才有。这还是最理想情况——谈代码量比时，你隐含假设弱语言里写得出来；但程序员能力有限。用太低级语言解难题，会到脑子里装不下的地步。

我说 ITA 假想对手要五年复制 ITA 三个月 Lisp 能写的东西，指一切顺利也要五年——多数公司里，任何要五年的项目大概永远做不完。这是极端案例：ITA 黑客异常聪明，C 又相当低级；但竞争激烈的市场里，二比一、三比一的差距就够让你永远落后。

### 行业最佳实践

这是尖头老板连想都不愿想的可能——所以多数不想。归根结底，尖头老板不介意公司被打得落花流水，只要没人能证明是他的错。对他个人，最稳妥是紧贴羊群中心。

大组织里这叫「行业最佳实践」——让尖头老板免责：选的是「行业最佳实践」，公司输了不能怪他；不是他选的，是行业选的。[5] 这词原指会计方法等：大致意思是，别做出格的事——在会计里大概是好主意；「前沿」与「会计」不配。但把这标准搬进技术决策，答案就错了。

技术往往该是前沿的。编程语言里，如 Erann Gat 指出，「行业最佳实践」给你的不是最好，而是**平均水平**。决策让你以激进对手几分之一的速度开发软件时，「最佳实践」用词不当。

两条极有价值的信息——我从经验知道属实：**一，语言能力不同；二，多数管理者故意忽略这点。** 两者合起来就是赚钱的配方。ITA 是实例：想在软件业赢，挑你能找到的最难问题，用最强大的语言，等对手的尖头老板回归均值。

### 附录：累加器生成器

说明语言相对能力，考虑：写函数生成**累加器**——接受数 n，返回函数；后者接受 i，返回 n **按 i 递增**（是递增，不是相加；累加器必须累积）。

Common Lisp：

```lisp
(defun foo (n)
  (lambda (i) (incf n i)))
```

Perl 5：

```perl
sub foo {
  my ($n) = @_;
  sub { $n += shift }
}
```

元素比 Lisp 多，因 Perl 须手动取参。

Smalltalk 略长：

```smalltalk
foo: n
| s |
s := n.
^[:i| s := s+i. ]
```

虽词法变量一般可用，不能对参数赋值，须新变量 s。

JavaScript 再略长——保留语句与表达式的区分，须显式 return：

```javascript
function foo(n) {
  return function (i) {
    return n += i
  }
}
```

（Perl 也保留这种区分，但以典型 Perl 方式允许省略 return。）

译成 Python 会撞墙：词法变量支持不全，须数据结构存 n；有函数类型但无字面量（除非函数体只是单个表达式），须命名函数返回：

```python
def foo(n):
    s = [n]
    def bar(i):
        s[0] += i
        return s[0]
    return bar
```

Python 用户会问为何不能 `def foo(n): return lambda i: return n += i` 或 `def foo(n): lambda i: n += i`——我猜总有一天也许能。（若不想等 Python 演化成 Lisp，他们总可以……）[4]

面向对象语言可在有限范围内用「单方法类 + 字段」模拟闭包（引用外围作用域变量的函数），等于让程序员做编译器在完整词法作用域语言里该做的分析；多函数共变量则不行，简单情形够用。Python 专家认可的写法：

```python
def foo(n):
    class acc:
        def __init__(self, s):
            self.s = s
        def inc(self, i):
            self.s += i
            return self.s
    return acc(n).inc
```

或：

```python
class foo:
    def __init__(self, n):
        self.n = n
    def __call__(self, i):
        self.n += i
        return self.n
```

我不愿 Python 拥护者说我歪曲语言，但两者在我看来都比第一版复杂——同样是为累加器单开位置，只是对象字段而非列表头；`__call__` 等保留字段名有点取巧。

Perl 与 Python 之争里，Python 方称 Python 是 Perl 的更优雅替代——此例说明**能力才是终极优雅**：Perl 程序更简单（元素更少），语法虽丑一点。

### 图灵等价与 Greenspun

Fortran、C、C++、Java、Visual Basic 能否解此题？不清楚。Ken Anderson 说 Java 最接近的是：

```java
public interface Inttoint {
    public int call(int i);
}
public static Inttoint foo(final int n) {
    return new Inttoint() {
        int s = n;
        public int call(int i) {
            s = s + i;
            return s;
        }
    };
}
```

未达规格：只对整数有效。与 Java 黑客多轮邮件后，写行为如前述例子的真正多态版本，介于极其别扭与不可能之间——有人写了我非常好奇，但我个人已放弃。

并非字面意义上别的语言解不了——它们图灵等价，严格说任何程序都能写。极限情况下：**在弱语言里写 Lisp 解释器。** 像玩笑，大项目里不同程度常发生，有名字：**Greenspun 第十条规则**——

> 任何足够复杂的 C 或 Fortran 程序，都含一个临时、非正式指定、bug 缠身、缓慢的 Common Lisp 半实现。

解难题时，问题不是会不会用够强的语言，而是：（a）用强语言，（b）写事实上的解释器，还是（c）自己当人肉编译器。Python 例已开始：我们在模拟编译器为词法变量生成的代码。

此做法不仅普遍，且已制度化。面向对象世界常谈「模式」——我怀疑模式有时是否是（c）人肉编译器在工作的证据。程序里见模式，我视为麻烦征兆：程序形状应只反映要解的问题；代码里别的规律性，至少对我，说明抽象不够强大——往往是我手动生成某宏该做的展开。[6]
